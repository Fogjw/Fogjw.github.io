MiniDB：从零实现一个数据库内核——项目开发日记

发表于2026-07-22|更新于2026-07-22|项目日记
|浏览量:

此文章用于记录在开发MiniDB项目中碰到的问题及解决方法，不懂的点和学习后的总结等诸多碎碎念。

项目介绍

  1. 定位：从零实现一个轻量级关系数据库，具备完整的数据库内核能力——缓冲池、索引、事务、恢复、SQL 支持。

  2. 目标：在项目开发完成时，此项目应具备能力：

  • 支持部分SQL语句（SELECT/INSERT/CREATE TABLE/DELETE）
  • 支持并发多线程读写，MVCC 快照分离
  • WAL 预写日志 + ARIES 崩溃恢复
  • 有 Web 管理面板（执行 SQL、看引擎监控）
  1. 技术栈：
  • C++17：模板、智能指针、RAII、移动语义
  • 并发编程：std::thread、std::shared_mutex、std::atomic、std::condition_variable
  • 存储引擎：B+Tree、Buffer Pool、Slotted Page、Clock-Sweep 淘汰
  • 事务与恢复：MVCC、Lock Manager、ARIES WAL
  • 网络通信：Unix Socket、Protocol Buffers
  • Web 开发：Go net/http、React + TypeScript
  • Linux 系统调用：open / pread / pwrite / fsync
  1. 项目地址：暂时私有

目前已经开发完成了两个模块（page和buffer），这里做一下总结

page 模块

设计思路

考虑到持久化记录，键值存储是一定要存入磁盘的，也会需要从磁盘中读取，而我们都知道，磁盘的读写速度，尤其是随机读写速度，与内存读写相比差别过大，为了减少磁盘对性能的阻碍，必须在这方面下功夫。

考虑到操作系统与内存交互的单位是页，因此考虑设计一种数据结构存储数据page，使得每次从磁盘中读取一个page后，一个page刚好分配一个内存页，使得分配到的OS页没有外部碎片浪费，同时减少磁盘I/O次数，写操作同理。同时根据时空局限性，以页为单位读到内存，后续访问的数据也可能是页中的，增加了缓存命中率。因此本项目中设计与磁盘交互的存储单位为page，每个page占4096字节。

而对于一个page，读取到内存之后，对于其中的4096字节空间，这里做了一些设想，大致如下图

Slot存放指向对应的数据record的指针（即相对可用空间最低地址的偏移量）和数据长度。
页头存放当前页的一些信息，如页id，剩余可用空间，slot数量等

遇到的问题

在设想中，数据的长度是不一的，也正因此设计了Slot定长结构来便于定位数据，取得slot中的数据偏移和长度就能进行随机访问。因此Slot必须能通过Slot的id快速定位，有两种方式实现。

  1. SlotId存入Slot中，由于Slot定长，所以支持从可用空间最高地址向下遍历
  2. Slot从可用空间最高地址紧密排列，从占据最高地址的Slot向低地址依次id为0，1，2…，不显式存储，靠位置计算

考虑如果要删除一条数据，对于数据部分，两种方式都一样，即要么每次删除都让删除数据的高地址方向的数据移动覆盖以消除内部碎片，要么定期调用整理方法消除内部碎片。但是对于Slot部分，方式1可以将剩余空间端的Slot移动覆盖被删除的数据对应Slot，消除Slot的内部碎片，如下图

而方式2由于SlotId由位置推断，如果移动则会导致混乱，因此只回收数据部分，对应Slot仅将字段重置不指向任何数据，这会导致产生一部分的内部碎片。

经过权衡，选择方式2实现，理由如下

  1. 方式2的Slot实现仅4B（方式1需要加id字段，总计6B），虽然会产生内部碎片，但每次删除数据也只会产生4B的内部碎片，开销并不大
  2. 方式2的Slot实现使根据SlotId获取Slot的效率高得多，可以直接计算出偏移量得到Slot指针，查找时间为常数级，而方式1每次查找都需要遍历，时间开销太大

Buffer 模块

这个模块主要是用于完成缓存池，考虑数据库数据大时，频繁访问数据库必然出现内存占用高的情况，此时操作系统会进行换页，但操作系统的换页开销不可控，会换到磁盘中不确定的位置，因此缺页异常换回页时开销也不稳定，且操作系统没有脏页概念，脏页换出后需要写入数据库文件需要再换回内存，另外也难以统计命中率（以为在缓存中，但实际上被换到了磁盘上）。

缓存池限制读到内存中的页数量，只在页不被使用时换出，且换出时将脏页写回文件，能对内存中的页做精细化策略控制，缓存池自行控制留在内存中的页。

设计思路

考虑到这是一个多线程并发访问的数据结构，缓存池必须设计原子操作和互斥锁。考虑到读操作应该允许同时访问，缓存池的锁均为读写锁（在Linux上，std::shared_mutex是偏向写者的读写锁）。缓存池的换页策略设计为Clock-sweep，即当缓存页命中时，设置一个标记，需要换页时循环遍历数组，寻找符合换出条件的页，若符合但有标记，则去掉标记继续循环遍历直到找到符合条件且无标记的页。

缓存池将页封装为一个新的数据结构Frame，增加是否为脏页、Clock-sweep标记、正在使用页的线程数、每个页自己的读写锁等属性，同时用std::vector容器存放Frame，使用std::unordered_map存放页id到Frame在vector容器中的索引的映射，同时用一把shared_mutex保护这个映射表。

遇到的问题

如果需要从磁盘中加载一页到内存，或新建一个空白页，都需要从Frame数组里取出一个Frame（根据策略），若为脏页就写入磁盘，然后从pid-frame_idx映射表中删掉映射，返回这个空闲下来的frame_idx来存放内存。但是初始化未加载的Frame中的pid为0，如果直接按Frame中的pid删掉映射，可能此时有pid为0的页加载在其他的Frame，导致误删实际上在内存中的映射。因此在淘汰一页返回空闲索引方法中加入了对初始页的判断逻辑。同时学到了一个C++的细节：

Frame的Page属性中的PageHead的pid并没有显式初始化为0，但是在调用vector<Frame> frames_(pool_size_)构造时，依然会把pid初始化为0，而不会是垃圾值。这种情况叫值初始化。因为Frame作为自定义结构体没有用户自定义的构造函数时，编译器会默认生成，先将整个对象初始化为0，再调用默认构造，将显式初始化的字段进行初始化。
